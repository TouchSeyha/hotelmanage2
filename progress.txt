## Codebase Patterns
- Prisma enums must match Zod schemas in src/lib/schemas/api/ for type safety
- After adding enum values to schema.prisma, run `bun db:push` to regenerate Prisma client
- StatusBadge component at src/components/shared/statusBadge.tsx must be updated when adding new status values
- tRPC mutations use adminProcedure for admin-only endpoints (from ~/server/api/trpc)
- Database transactions use ctx.db.$transaction([...]) for atomic operations
- BookingLog entries should be created for all significant booking state changes
- Quick actions UI pattern: dialog confirmation before executing mutations

### Learnings for future iterations
- **Enum sync pattern**: When adding new values to Prisma enums, must update:
  1. prisma/schema.prisma enum definition
  2. Corresponding Zod schema in src/lib/schemas/api/
  3. StatusBadge component type unions and config objects
  4. Run `bun db:push` to regenerate Prisma client
  5. Run `bun typecheck` to catch any type mismatches
- **Transaction pattern**: Use ctx.db.$transaction([]) to ensure booking status, room status, and log entries are updated atomically
- **Quick actions pattern**: The quick-actions.tsx component follows a consistent pattern:
  - State for dialog visibility (e.g., showEarlyCheckOutDialog)
  - Mutation hook with onSuccess invalidation
  - Handler function that calls mutation
  - Button with loading state
  - ConfirmDialog component with clear description
- **Admin pages location**: Admin pages are in src/app/(admin)/admin/ directory structure
- **Existing functionality**: Most of the early checkout feature was already implemented, only needed to:
  - Add the missing enum values to schema
  - Update TypeScript types to match
  - Regenerate Prisma client
- **Status workflow**: The booking status flow is: pending -> confirmed -> checked_in -> checked_out (early) OR completed (normal checkout)
- **Room status workflow**: Room goes from occupied -> cleaning (early checkout) or available (normal checkout)
